# 系统架构图（Architecture Diagram）

## 这是什么图

展示系统的整体架构、服务分层、组件依赖关系和数据流向的图表。适合描述微服务架构、云原生系统、分布式系统的拓扑结构。

## 适用场景

- 微服务架构设计与评审
- 云原生系统部署拓扑
- 系统技术方案文档
- 新成员 onboarding 时的系统全貌介绍
- 系统重构前的现状梳理

## 你怎么描述

用自然语言回答以下问题（不需要全答，AI 会追问你漏掉的部分）：

1. **系统叫什么名字？**
   比如："电商平台"、"支付系统"、"物联网数据平台"、"用户中心"

2. **系统分几层？每层有哪些组件？**
   不要求固定层数，按你的实际情况描述：
   - 前端层：Web 端（React/Vue）、移动端（iOS/Android）、小程序
   - 网关层：API 网关（Kong/Nginx/Spring Cloud Gateway）、负载均衡
   - 服务层：各微服务模块（用户服务、订单服务、商品服务……）
   - 中间件层：消息队列（RabbitMQ/Kafka）、缓存（Redis）
   - 数据层：MySQL、PostgreSQL、Elasticsearch、MongoDB
   - 1-6 层都可以，取决于你的系统复杂度。

3. **组件之间怎么连接和交互？**
   比如：
   - "客户端请求经过 API 网关，网关分发到各微服务"
   - "订单服务通过 RabbitMQ 异步通知库存服务"
   - "用户服务和订单服务各自有独立的 MySQL 数据库"
   需要明确谁调谁、同步还是异步、数据怎么流动。

4. **有没有特殊要求？**
   比如：
   - "用蓝色标注核心服务，红色标注安全相关组件"
   - "把即将下线的模块用虚线框出来"
   - "需要在图上标注技术栈（Go、Java、Python）"
   - "高亮用户服务——它是整个系统的核心入口"

## 示例

### 示例1：简单的三层架构

> 画一个电商系统的架构图。前端有 Web 端和 iOS App，通过 Nginx 网关访问后端。后端有用户服务和订单服务两个微服务。每个服务有自己的 MySQL 数据库。还有一个 Redis 做缓存。

### 示例2：详细的微服务架构

> 画一个电商平台的微服务架构图。
>
> 系统名称：电商平台
>
> 前端层：
> - Web 端（React，部署在阿里云 CDN）
> - iOS App（Swift）
>
> 网关层：
> - API 网关用 Kong，负责路由、限流、鉴权
>
> 服务层：
> - 用户服务（Go 语言）：注册、登录、JWT 鉴权、用户信息管理
> - 订单服务（Java / Spring Boot）：下单、查订单、对接支付宝支付
> - 商品服务（Python / FastAPI）：商品 CRUD、商品搜索用 Elasticsearch
> - 库存服务（Go 语言）：扣库存、预占库存、库存同步
>
> 中间件层：
> - RabbitMQ：订单服务的异步消息（订单创建 → 库存扣减）
> - Redis：用户 session 缓存、商品热点数据缓存
>
> 数据层：
> - 每个微服务有独立的 MySQL 数据库
> - 商品搜索使用 Elasticsearch
>
> 部署环境：
> - 整个系统部署在阿里云 Kubernetes（ACK）上
> - 生产环境 3 节点集群，每个服务 2 副本

## 约束（AI 生成时必须遵守）

- **布局方向**：从上到下分层排列。层与层之间间距 ≥ 120px。
- **同层排列**：同一层的组件水平间距 100–120px，组件之间边缘间距 ≥ 80px。
- **颜色语义**：严格按照语义角色分配颜色——
  - 服务/应用组件：蓝色（fillColor=#dae8fc, strokeColor=#6c8ebf）
  - 数据库/存储：绿色（fillColor=#d5e8d4, strokeColor=#82b366）
  - 消息队列/异步：黄色（fillColor=#fff2cc, strokeColor=#d6b656）
  - API 网关/代理：橙色（fillColor=#ffe6cc, strokeColor=#d79b00）
  - 安全/鉴权组件：红色（fillColor=#f8cecc, strokeColor=#b85450）
  - 外部系统/第三方：灰色（fillColor=#f5f5f5, strokeColor=#666666）
  - 配置/基础设施：紫色（fillColor=#e1d5e7, strokeColor=#9673a6）
- **网关位置**：API 网关放在顶层中央，数据库放在底层。
- **服务容器**：同一微服务内的子组件用 swimlane 容器包裹，容器内子组件上下排列。
- **走线通道**：层与层之间至少保留 80px 宽的走线通道，避免边线穿越组件。
- **Z 序**：边在组件下面渲染（边的 id 编号小于顶点）。
- **坐标对齐**：所有坐标（x, y, width, height）必须是 10px 的整数倍。
- **边规范**：边的 `<mxGeometry>` 必须展开（不能自闭合）。
- **标题**：图的最上方添加标题，fontSize=14, fontStyle=1（加粗）。
