# 时序图（Sequence Diagram）

## 这是什么图

展示多个参与者之间的消息交互顺序，按时间从上到下排列。适合描述 API 调用链、登录流程、下单流程、分布式事务等场景。

## 适用场景

- 用户登录/注册流程设计
- 订单处理与支付流程
- 微服务间 API 调用链分析
- 分布式事务（Saga）流程
- 第三方接口对接时序
- OAuth 2.0 / SSO 认证流程

## 你怎么描述

用自然语言回答以下问题（不需要全答，AI 会追问你漏掉的部分）：

1. **有哪些参与者（角色）？**
   比如："用户、客户端（App/浏览器）、API 网关、用户服务、数据库"
   参与者就是图顶部那些方框，一般 3–6 个比较合适。太多会显得拥挤。

2. **交互流程是什么样的？**
   按时间顺序列出消息，格式：**谁 → 谁：消息内容**
   例如：
   1. 用户 → 客户端：点击"登录"按钮
   2. 客户端 → API 网关：POST /api/login
   3. API 网关 → 用户服务：转发登录请求
   4. 用户服务 → 数据库：SELECT user WHERE email=?
   5. 数据库 → 用户服务：返回用户记录
   6. 用户服务 → API 网关：返回 JWT Token
   7. API 网关 → 客户端：200 OK + JWT
   8. 客户端 → 用户：显示登录成功

3. **哪些是请求，哪些是响应/返回？**
   - 请求（同步调用）：实线箭头，表示主动调用
   - 响应/返回：虚线箭头，表示返回结果
   - 异步消息：虚线箭头，标注"异步"

4. **有没有特殊标注？**
   比如：
   - "需要标注数据库查询的具体 SQL"
   - "在关键步骤上标注超时时间（如 3s timeout）"
   - "高亮认证相关的步骤"

## 示例

### 示例1：简单登录流程

> 画一个用户登录的时序图。参与者：用户、客户端 App、API 网关、用户服务、数据库。
>
> 流程：
> 1. 用户打开 App，输入账号密码，点击登录
> 2. 客户端 POST /api/login → API 网关
> 3. API 网关转发 → 用户服务
> 4. 用户服务 SELECT 查询 → 数据库
> 5. 数据库返回用户数据 → 用户服务
> 6. 用户服务验证密码，生成 JWT → API 网关
> 7. API 网关返回 JWT → 客户端
> 8. 客户端存储 Token，跳转主页
>
> 返回消息（4、5、6、7）用虚线箭头。

### 示例2：电商下单流程

> 画一个电商下单的完整时序图。
>
> 参与者：用户、Web 前端、订单服务、库存服务、支付服务（支付宝）、消息队列（RabbitMQ）。
>
> 流程：
> 1. 用户 → Web 前端：点击"立即购买"
> 2. Web 前端 → 订单服务：POST /orders（商品ID、数量、地址）
> 3. 订单服务 → 库存服务：POST /inventory/reserve（预占库存）
> 4. 库存服务 → 订单服务：200 OK（预占成功，返回预占ID）
> 5. 订单服务 → Web 前端：返回订单ID + 支付链接
> 6. Web 前端 → 用户：跳转支付宝收银台
> 7. 用户 → 支付宝：输入支付密码
> 8. 支付宝 → 订单服务：POST /orders/{id}/payment-callback（异步回调）
> 9. 订单服务 → 库存服务：POST /inventory/confirm（确认扣减库存）
> 10. 库存服务 → 订单服务：200 OK
> 11. 订单服务 → 消息队列：发布"订单已支付"事件
> 12. 订单服务 → Web 前端：推送"支付成功"通知
> 13. Web 前端 → 用户：显示"下单成功"
>
> 请求用实线箭头（1、2、3、7、9、11），返回和异步消息用虚线箭头（4、5、8、10、12）。

## 约束（AI 生成时必须遵守）

- **参与者排列**：参与者矩形放在顶部（y=40, height=40），从左到右排列，水平间距 150–200px。
- **生命线**：每个参与者下方画一条垂直虚线（dashed=1;dashPattern=8 4;），从参与者底部延伸到 y=500（或最后一条消息以下 80px）。
- **消息递增**：每条消息的 Y 坐标递增 50px（第一条 y=130，第二条 y=180，以此类推）。
- **消息方向**：请求消息（实线）从左到右，返回消息（虚线）从右到左。
- **箭头样式**：
  - 请求消息：实线、实心箭头（endArrow=classic;endFill=1;）
  - 返回消息：虚线、空心箭头（dashed=1;endArrow=open;endFill=0;）
- **定位方式**：消息边使用 `sourcePoint` / `targetPoint` 精确定位在生命线上，X 坐标必须与对应参与者的生命线 X 坐标一致。
- **边标签背景**：所有带文字的消息边必须添加 labelBackgroundColor=#FFFFFF。
- **坐标对齐**：所有坐标必须是 10px 的整数倍。
- **参与者上限**：单页不超过 6 个参与者。
